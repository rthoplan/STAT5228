Tidy Data

The layout that makes analysis easy, predictable, and scalable.

1 The Three Core Rules

Variables as Columns

Each column is a unique variable. Avoid having multiple columns representing the same attribute (such as "Sales_2022" and "Sales_2023").

Observations as Rows

Each row is one specific "event" or "unit." If you measured a person three times, they should have three distinct rows.

One Table per Unit

Don't mix observational levels. For instance, don't mix "Store Data" (address) with "Transaction Data" (items bought) in one table.

2 Visualising the Difference

Untidy (Wide Format)

City Morning_Temp Evening_Temp
London 15°C 12°C

The Issue: Time of day is trapped in column names. To compare temperatures, you'd have to write specific code for "Morning_Temp" and "Evening_Temp" separately.

Tidy (Long Format)

City Time Temp
LondonMorning15°C
LondonEvening12°C

The Fix: Time is now a variable. You can "group by Time" to find the average morning temperature across all cities without extra effort.

3 Understanding "Special Casing"

"If you have to name specific data values in your code, it's not tidy."

"Special Casing" occurs when your data structure forces you to write code that only works for a fixed set of columns. You are treating each column as a "special case" rather than treating a column as a variable that holds many values.

Special Casing (Manual)

Your code must "know" the names of every month in the headers.

// You manually list every "Case"
total = data['Jan'] + data['Feb'] + data['Mar']

This is "Special Casing" because you are hard-coding 'Jan', 'Feb', and 'Mar'. If 'April' appears, your code is unaware of it.

General Logic (Automatic)

Your code only needs to know the variable name Revenue.

// One rule for all months
total = data['Revenue'].sum()

This works for any number of months automatically. You are referring to the variable (Revenue), not the data (Jan/Feb/Mar).

Summary: If you find yourself typing column_name_A + column_name_B, you are special casing. If you can type sum(variable_name), your data is tidy.

Why Tidy Data Helps

Consistent Inputs

Plotting and modelling functions expect one column to represent one "aesthetic" (e.g., colour or x-axis). Tidy data aligns perfectly with these toolsets.

Simpler Wrangling

Grouping, filtering, and aggregating become single-command operations. You no longer need to write complex loops to navigate through headers.

Reproducible Pipelines

A tidy pipeline (wrangle → plot → report) is robust. You can update your dataset with new dates or categories, and the analysis will run without modification.

Fewer One-Off Cases

By following the three rules, you eliminate the need for "special casing" logic, reducing the number of bugs in your code.

Tidy data isn't just about "neatness"—it's about making your analysis immune to change.
